# digest/ — analytics, weekly report, email digest, shared presentation

Deep docs for `digest/analytics.py`, `digest/weekly_report.py`,
`digest/email_digest.py`, `digest/presentation.py`. Loads when working in
`digest/`. High-level overview is in the root `CLAUDE.md`. (Note:
`compute_skill_gaps` lives in `digest/presentation.py` but is documented in
`scorer/CLAUDE.md` since it feeds training recs.)

## Analytics + weekly career report (`digest/analytics.py`, `digest/weekly_report.py`)
Pure aggregation, no LLM call and no new API usage — reads only `storage.tracker_store.load_tracker()` (and, for the weekly report, `storage.run_store.list_runs()`), the same zero-cost-aggregation pattern as `digest.presentation.compute_skill_gaps`. `digest/analytics.py` exposes `resume_performance()` (interview/applied conversion grouped by `chosen_archetype`), `score_band_conversion()` (conversion rate by `last_score` bucket), `top_missing_keywords(statuses=("rejected",))` (frequency-ranked, same pattern as the skill-gap report but scoped to a status), `application_summary()`, and `company_intelligence(top_n=10)` (per-company avg score / applied / interviewed / interview rate) — all tolerate an empty tracker, and all exclude ghost-flagged tracker entries. `digest/weekly_report.py`'s `build_weekly_report()` stitches these into one markdown report (this week's runs by source, applications, resume performance by framing, most responsive companies, top missing keywords in rejections, recommended actions) and writes it to `data/weekly_report_<date>.md` (git-committed, like `data/runs/` and `data/tracker.json`). It only runs when `is_report_day()` is true — a Friday, matching the pipeline's weekday schedule — checked once near the end of `main.run()`, after the mid-run tracker update so the report reflects this run's fresh data. `digest/email_digest.py`'s `_weekly_report_section()` renders the same markdown into an HTML block (a small hand-rolled converter, no new dependency) appended to that day's digest email.

## Digest (`digest/email_digest.py`)
Builds one HTML email split into three sections by `is_target_location`/`is_qa_role`: **primary** (target location + non-QA — full match table + per-job recommendations + skill-gap summary + "Recommended Training" + "Fast-Apply Packs" with cover-note variants/screening answers), **outside target location**, and **QA / Testing Roles** (the latter two are informational-only — table with score/seniority/industry/missing-keywords/apply-link, no Resume column, since `main.py` never spends Gemini optimize quota on them). `main.py` builds these as three separate lists (`primary`, `outside_target_location`, `qa_roles`), each sorted by `match_score` descending then `seniority_fit` (`good` > `over` > `under`), and passes all three plus `profile` into `_build_html`/`send_digest`. Attaches up to `TOP_ATTACH_COUNT` (15) ATS resumes by score, drawn only from `primary`. It deliberately never auto-submits anything (job-board ToS risk) — the Fast-Apply Pack is copy-paste material the user applies manually. Includes a `mailto:` "Leave Feedback" link pre-filled with a message meant to be pasted into a future Claude Code session to drive the next fix.

Within `primary`, `_build_html` further splits jobs by `resume_track` (a tag `main.py` only sets when `profile.secondary_resume_path` — a second, differently-focused résumé — is configured and that résumé's keyword ranking is what surfaced the job; see `scorer/CLAUDE.md`). `_build_primary_section(jobs, track_label)` builds one track's table + recommendations + Fast-Apply Packs; on a day with no `resume_track: "secondary"` jobs it's called once with no label and renders exactly as a single-resume digest always has, so this is purely additive. When both tracks have jobs, it renders twice under `profile.primary_track_label` and `profile.secondary_track_label` headings, each with its own table and Fast-Apply Packs (tailored from that job's actual matching resume, not the default one).

## Shared presentation logic (`digest/presentation.py`)
`score_color`, `score_bg`, `seniority_badge_colors`, `industry_badge_colors`, `industry_label` map a job's `industry_fit` value to a display label and badge color. Since `profile.industry_bands` can define any number of bands with any `value`/`label`, `industry_badge_colors` colors by the band's **rank** within `profile.industry_bands` (best → worst), not by hardcoded literal values, and `industry_label` looks up the configured `label` for that `value` (falling back to a title-cased version of the raw value if no `industry_bands` context is passed — the dashboard's Jinja2 templates currently render badges without per-request profile context, so this fallback keeps them from breaking). These were extracted out of `digest/email_digest.py` into pure functions so the HTML email and the dashboard's templates render identical badges without drift. Jobs whose company is on `profile.target_companies` (`scraper._is_target_company` → `target_company`) get a "⭐ target company" highlight (soft row tint in the digest table, ⭐ on the dashboard) — presentation-only, no effect on score or ordering. `email_digest.py` keeps thin wrapper functions (`_seniority_badge`, `_industry_badge`) that build the same inline-style `<span>` markup as before — its HTML output is unchanged.
