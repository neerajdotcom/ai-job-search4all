# webapp/ — local-only FastAPI + Jinja2 dashboard

Deep docs for `webapp/` (`app.py`, `run_manager.py`, `__main__.py`). Loads
when working in `webapp/`. High-level overview is in the root `CLAUDE.md`.
Persistence internals (`data/runs/`, `data/tracker.json`) live in
`storage/CLAUDE.md`.

A FastAPI + Jinja2 dashboard for viewing run history/job matches and triggering runs from a browser. **Local-only by design** — bound to `127.0.0.1`, no auth layer, never deployed to a hosted PaaS (free hosting tiers there sleep/expire; running it on your own machine is the only option that stays free forever). It reads/writes the exact same repo clone the CLI and `git pull` use, so it works whether you're looking at runs from your own machine or runs that happened in GitHub Actions.

- **Run:** `pip install -r requirements.txt` then `python -m webapp` → open `http://127.0.0.1:8000/`.
- **Pages:** `/` (summary cards + Chart.js charts: score distribution, qualifying-jobs-over-time, source breakdown, top companies), `/runs` (paginated run history table; the currently-running webapp-triggered run, if any, is pinned as a highlighted in-progress row at the top of page 1, identified by a run ID assigned at run *start* — non-clickable until its snapshot is written — and kept live by `initRunsPage()` polling `/api/runs/status`, which reloads the page when the run finishes so the new snapshot joins the completed rows), `/runs/{run_id}` (three job tables for one run — Primary Matches / Outside Target Location / QA & Testing Roles, split by each job's `is_target_location`/`is_qa_role` tags, with score/seniority/industry badges; `job_index` in each row's link is `loop.index0` over the *full* `run.jobs` list so it still lines up with `/downloads/...` and the job-detail route regardless of which section a row appears in), `/runs/{run_id}/jobs/{job_index}` (full job detail — keywords, recommendation, chosen framing archetype, an application-status dropdown (the full lifecycle vocabulary — `new`/`scored`/`qualifying`/`optimized` auto-set by the pipeline, `applied`/`interview_scheduled`/`interview_complete`/`offer`/`rejected`/`withdrawn`/`archived` settable here) writing the cross-run tracker, plus free-text `recruiter_name`/`notes`/`follow_up_date` fields, per-variant copy-to-clipboard cover notes, screening Q&A, resume download links, a "⚠ link may be closed" flag from the liveness check), `/trigger` (dry-run-checked-by-default form + an "Enable crawl4ai web-board scraping" checkbox, unchecked by default + live progress panel that polls run status), `/kanban` (a drag-and-drop application-pipeline board built entirely from `data/tracker.json` — no fabricated fields — via `tracker_store.kanban_board()`; columns are `qualifying → optimized → applied → interview_scheduled → interview_complete → offer`, with `rejected`/`withdrawn`/`archived`/legacy-alias entries shown in a separate non-draggable "Closed" row below the board; `new`/`scored` entries aren't shown since they aren't yet part of an application pipeline; cards link out to `apply_url` since tracker entries have no `run_id`/`job_index` to link back to an internal detail page).
- **Theme toggle:** a 🌙/☀️ button in the navbar (`initThemeToggle()` in `dashboard.js`) flips a `data-theme="dark"` attribute on `<html>` and persists the choice in `localStorage`; `style.css` defines every color as a CSS variable on `:root` with a `[data-theme="dark"]` override block reusing the same variable names, so every existing rule repaints with no per-rule dark-mode CSS. An inline script early in `base.html`'s `<head>` (before the stylesheet `<link>`) applies the saved theme before first paint to avoid a flash of the wrong theme. Chart.js reads colors once at construction time, not reactively, so `initHomeCharts()` tracks its instances in `homeCharts` and destroys/recreates all of them on every theme flip (and on first load) using a theme-aware palette (`LIGHT_PALETTE`/`DARK_PALETTE`/`currentPalette()`).
- **JSON API** (used only by the dashboard's own JS for polling/charts, not a public API): `POST /api/runs/trigger` (`dry_run`/`resume_path`/`no_dedup`/`enable_crawl4ai` in the body; `202` on start, `409` if a run is already active), `POST /api/tracker/status` (sets a posting's application status in `data/tracker.json` given `run_id`+`job_index`), `POST /api/tracker/status_by_key` (same status-set, but by tracker `key` directly — used by the Kanban board, which already has the key on each card and has no `run_id`/`job_index` to resolve), `POST /api/tracker/fields` (sets the manual `recruiter_name`/`notes`/`follow_up_date` bookkeeping fields on a tracker entry), `GET /api/runs/status`, `GET /api/stats`, `GET /api/runs`, `GET /downloads/{run_id}/{job_index}/{ats|review}` (serves the DOCX if it still exists on disk).

## Triggering a run from the dashboard (`webapp/run_manager.py`)
`main.run()` takes an optional `source` (defaults to auto-detecting `"github_actions"` vs `"cli"`; the dashboard explicitly passes `source="webapp"`) and an optional `progress_callback(update: dict)` invoked at each pipeline checkpoint (scraping, scraped, prefiltering, each scoring/optimizing iteration, sending digest, done) — wrapped in try/except so a bad callback can never crash the pipeline. The dashboard runs the pipeline in a single dedicated daemon thread (not asyncio, not a task queue) because the pipeline is fully synchronous blocking I/O with its own `time.sleep()`-based throttles; a `threading.Lock` + atomic check-and-set `active` flag in `RunManager.start_run()` enforces **one run at a time** (a second trigger while one's active gets `409`, not queued) — the pipeline's rate-limit/quota flags and `outputs/` directory aren't safe for concurrent runs. `start_run()` also assigns a `run_id` (via `run_store.run_id_for`) at run *start* so the in-progress run is identifiable on `/runs` before any snapshot exists; this provisional id can differ from the final `result_run_id`/snapshot filename by ~1s (the snapshot is named from `RunSummary.started_at`, created a moment later inside `main.run()`), which is cosmetic since the in-progress row is non-linking and is replaced on reload the instant the run finishes. Only webapp-triggered runs are visible as in-progress — CLI and GitHub Actions runs execute in separate processes and appear on `/runs` only once their snapshot is committed and pulled.

`_run_worker()` also sets `os.environ["ENABLE_CRAWL4AI"]` from the trigger form's checkbox right before calling `main.run()`, explicitly to `"true"` or `"false"` (never left unset) so the flag never leaks from one run's checkbox state into the next — `scraper/job_scraper.py` reads this same env var (see `scraper/CLAUDE.md`). This is the one supported way to opt a *single* webapp-triggered run into the optional crawl4ai web-board sources (Naukri/Indeed/Foundit/Instahyre/Wellfound); they're disabled in CI and off by default here because most of those boards block datacenter IPs — check it only when running the dashboard from a residential IP.

## Resume download graceful degradation
There's no stored "ran in CI vs locally" flag. `/runs/{run_id}/jobs/{job_index}` and the `/downloads/...` route both just check `Path(ats_output/review_output).exists()` at render time and again right before serving the file. This naturally produces three different outcomes with zero extra bookkeeping: a GitHub-Actions-run snapshot viewed locally → file never existed on this machine → "not available, check the Actions artifact"; a webapp-triggered run viewed locally → file exists → real download; a dry-run snapshot → `ats_output is None` → "not generated (dry run)".

## Tests
**No webapp-specific tests** — verify by running `python main.py --dry-run` (confirm a new `data/runs/<run_id>.json` with `source: "cli"` appears) and `python -m webapp` (browse the pages, trigger a dry run, confirm a second snapshot with `source: "webapp"` appears and `/` picks it up).
