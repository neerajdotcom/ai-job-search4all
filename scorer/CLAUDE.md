# scorer/ — prefilter, ghost detection, scoring, reviewer, training recs

Deep docs for `scorer/match_scorer.py` (plus the off-target skip and
skill-gap report that are tightly coupled to scoring). Loads when working in
`scorer/`. High-level overview is in the root `CLAUDE.md`.

## Pre-filter (`scorer.match_scorer.prefilter_jobs`)
Zero-cost local keyword ranking (resume/JD token overlap + priority terms derived from `profile.search_terms`/`profile.title`/`profile.adjacent_industries` + title bonus) that trims the job pool to `AI_SCORE_LIMIT` (50, set in `main.py`) before any quota-limited LLM call. Sorts by `(_priority_rank(job), -prefilter_score)`, not score alone — target-location+non-QA jobs (`_priority_rank` 0) always claim a scoring slot ahead of out-of-target-location jobs (rank 1) and QA/testing jobs (rank 2), so a foreign or QA posting with better keyword overlap can never bump a target-location role out of the scarce Groq-scoring cap. `main.py` runs the cross-run dedup gate (`storage.tracker_store.filter_unscored`, see `storage/CLAUDE.md`) *before* this prefilter, not after — ranking the full deduped pool, not just whatever the dedup gate happened to leave behind. (The reverse order was a real bug: prefilter-first meant the same top-50 jobs got reselected and re-skipped by dedup every run, starving the pipeline of new scoring candidates for days.) If `profile.secondary_resume_path` (an optional second, differently-focused résumé) is set, `prefilter_jobs` is called twice — once per résumé, each ranking only what the other pass hasn't already claimed — splitting `AI_SCORE_LIMIT` across both tracks instead of doubling Groq usage; jobs picked by the second pass are tagged `resume_track: "secondary"` so `main.py` scores/tailors them against that résumé instead of the default one, and `digest/email_digest.py` renders them as a separate group under `profile.secondary_track_label`.

## Ghost-job detection (`scorer.match_scorer.is_ghost_job`)
Four zero-cost local heuristics run on every prefiltered job *before* it reaches Groq, in the same spirit as `prefilter_jobs`/`_experience_override` — flags (a) a JD under 100 words, (b) an empty or placeholder company name (`unknown`/`n/a`/`confidential`), (c) a title matching urgency/spam language (`urgent`, `immediate joiner`, `walk-in`), or (d) a JD whose normalized text is duplicated across more than `_GHOST_DUPLICATE_THRESHOLD` (3) other companies' postings in the same scrape batch, via a fingerprint index (`build_jd_fingerprint_index`) built once per run over the in-memory job list — no extra fetch or API call. A flagged job is **never hard-dropped**: `main.py` tags it `ghost_flagged=True` with a `ghost_reason`, counts it in `RunSummary.ghost_flagged_count`, and skips straight past `score_job`/the reviewer pass/Gemini optimize — saving a scarce Groq scoring slot (and, transitively, an optimize slot) for a posting that was never going to be worth one — but it still appears in the run snapshot and the dashboard's run-detail table (a "👻 Ghost" pill with the reason as a tooltip) so the reasoning is visible. Ghost-flagged entries are excluded from `digest/analytics.py`'s aggregates so their placeholder `match_score: 0` doesn't skew averages.

## Off-target skip (in `main.py`'s scoring loop, right before the ghost-job check)
A zero-cost title check that skips a Groq scoring call entirely for any **ATS-sourced** job (`source != "adzuna"`) that is neither `is_target_role` nor `is_qa_role` — i.e. a posting from a company's Greenhouse/Lever/Ashby/Workable board that's plainly outside the candidate's target career tracks (as defined by `profile.search_terms`/`profile.title`). Adzuna jobs are exempted since they were already relevance-filtered at query time by `profile.search_terms`; QA roles are exempted since they still get scored for their own digest section. Like ghost-job detection, this is a skip, not a drop: `main.py` tags the job `off_target_skipped=True` with a placeholder `match_score: 0`, counts it in `RunSummary.off_target_skipped_count`, and it still appears in the run snapshot and dashboard run-detail table (a "🎯 Off-target" pill). This exists because ATS feeds return a company's **entire** job board unfiltered by title (see `scraper/CLAUDE.md`) — without this gate, an unrelated role can consume one of the scarce `AI_SCORE_LIMIT` Groq scoring slots for zero benefit, since it has no qualifying path and no dedicated digest section either way. Off-target-skipped entries are excluded from `digest/analytics.py`'s aggregates, same as ghost-flagged ones.

## Scoring (`scorer/match_scorer.py`) — profile-driven merit rubric
Runs on **Groq** (`llama-3.3-70b-versatile`), not Gemini — chosen for its much higher free-tier request volume, since every prefiltered job gets scored. The prompt (`build_scoring_instructions(profile)`/`PROMPT_TEMPLATE`) computes `match_score` as the **sum of three merit categories out of 100**: SKILLS (from `profile.skill_areas`, a list of `{label, max_points}` bands that together sum to 40 — a generic single "skills and tools" band by default), INDUSTRY (from `profile.industry_bands`, pick the single best-matching band out of up to 35 points — defaults to a 3-tier core/adjacent/unrelated split at 35/18/6), and ROLE LEVEL (from `profile.role_level_bands`, up to 25 points — defaults to a 2-tier at-target/below-target split at 25/15). `industry_fit` is whatever set of `value`s `profile.industry_bands` defines; the digest/dashboard badge coloring (`digest/presentation.py`) ranks these bands best-to-worst rather than hardcoding specific values, so it works for however many bands a profile configures. The prompt also tells the model to judge the role's actual function/domain, not shared job-title words, using `profile.adjacent_industries` to distinguish a genuinely adjacent role from an unrelated one. **Target-company membership is NOT part of this score** — it's a separate prioritization signal (see below), so any company not on `profile.target_companies` can still reach 100 on merit. `SCORE_THRESHOLD` in `main.py` (60) is the single canonical qualification gate.

Before calling Groq, `_experience_override` extracts a required-years-of-experience figure (snippet first, falling back to the full page) — Adzuna's snippet is truncated and an LLM scoring only the snippet can't see a "7+ years required" clause further down. The extractor (`_extract_required_years`) matches both the full word and common abbreviations (`years`/`yrs`, plus `exp` alongside `experience` in the proximity guard), and reads a **range** ("10-14 years", "10 to 14 yrs") at its *lower* bound, so a "5-8 years" role the candidate (`profile.years_experience`) already meets isn't mistaken for an 8-year requirement. Any role requiring more years than `profile.experience_exclude_years` is **flagged and soft-capped to 45** (`skip_scoring`: Groq call skipped to save quota) — below `SCORE_THRESHOLD` so it never lands as a false digest match, but kept in the run snapshot/dashboard with a `⚠ Requires N+ years` reason for the user to judge (flag-not-drop, like ghost/off-target).

## Target-company highlight (curated, presentation-only) — `scraper.job_scraper._is_target_company`
`profile.target_companies` (a list you maintain) tags each job `target_company` (bool) at scrape time, beside `is_qa_role`/`is_target_location`/`is_target_role`. It has **no effect on scoring, qualification, or ordering** — it drives only a visual highlight (a "⭐ target company" marker + soft row tint in the digest, a ⭐ on the dashboard) so a job from a company you care about catches the eye. A miss or rare false-match is purely cosmetic. Short/ambiguous company names should use the fuller company name as the alias and are matched as a whole token to avoid false hits.

## Drafter-reviewer second pass (`scorer.match_scorer.review_score`)
For jobs that already cleared `SCORE_THRESHOLD` but landed in the borderline band (`SCORE_THRESHOLD` to `REVIEW_BAND_MAX` (75), set in `main.py`), `main.py` makes one more Groq call to sanity-check that the first pass's score is actually consistent with its own stated matched/missing keywords and recommendation — catching cases like a high score sitting next to several serious missing requirements, or a high score driven only by generic vocabulary when the role's real domain/function is unrelated to the candidate's background. Jobs already scoring `>= REVIEW_BAND_MAX` skip the call entirely — a clearly-strong match isn't worth auditing, and most qualifying jobs land well above the threshold, so this band check cuts the reviewer pass's token cost roughly in half versus reviewing every qualifying job. Deliberately does **not** resend the resume or full JD (only a 400-char excerpt plus the first pass's own compact output) to keep the added cost small — roughly +500 tokens per reviewed job, vs. ~1,500-2,000 tokens for the original full-context scoring call. If the reviewer's `reviewed_score` drops back below `SCORE_THRESHOLD`, the job is demoted (`qualifying` flips to `False`) before any Gemini optimize quota is spent on it. Fails open on any error or exhausted daily quota — `review_score` returns `None` and the first-pass score stands unchanged. Stored on the job as `reviewer_score` / `reviewer_notes`.

## Skill-gap report (`digest.presentation.compute_skill_gaps`)
Zero-extra-LLM-call aggregation that frequency-ranks every scored job's `missing_keywords` for the run, surfacing which skills show up most often as gaps across that batch. Computed once at the end of `main.run()` from `on_track = [j for j in all_scored if j.get("is_target_role")]` — **not** the full `all_scored` list — so an unrelated ATS-board posting (a role that nonetheless cleared the score threshold on generic keyword overlap) can't inject irrelevant "missing keywords" into the candidate's aggregate (this is what feeds the digest's "Recommended Training" section). Attached to the run snapshot (`skill_gaps`) and the digest email ("Skill Gaps This Run" section); also rendered on the dashboard's run detail page. (The function physically lives in `digest/presentation.py` but is documented here because it feeds training recs and is scoped by the scorer's `is_target_role` tag.)

## Training recommendations (`scorer.match_scorer.generate_training_recommendations`)
One **Groq** call **per run** (not per job) that turns the `skill_gaps` list above into 2-3 concrete course/cert picks (`{skill, recommendation, why}`). Runs on Groq (already initialized, abundant per-day budget) rather than Gemini (whose per-call throttle/optimize cap is scarcer), reuses the same `_generate_with_retry`/fence-stripping/`json.loads` pattern as scoring, and fails open to `[]` on any error or exhausted quota (digest just omits the section). Attached to the run snapshot (`training_recs`) and rendered in the digest ("Recommended Training") and the dashboard run-detail page.

## Gotchas
- Both `match_scorer.py` and `resume_optimizer.py` carry a `_daily_quota_exhausted` flag, set the first time Groq/Gemini reports the *daily* cap is hit (not a transient per-minute rate limit). Once set, the rest of that run skips calling that provider entirely instead of retrying every remaining job with backoff and failing — without this, a quota hit partway through a run (e.g. job 20 of 50) used to burn 20+ extra minutes retrying doomed calls on every job after it.
- `_MIN_INTERVAL` in `match_scorer.py` is the per-provider throttle for Groq; `resume_optimizer.py` has its own. When changing models or call volume, adjust these and the caps in `main.py` (`AI_SCORE_LIMIT`, `AI_OPTIMIZE_LIMIT`) rather than removing them.
- `_QA_TITLE_PATTERNS`-driven QA routing (see `scraper/CLAUDE.md`) is a legacy carve-out from the original candidate this tool was built for. It's harmless to leave in place for a profile whose `search_terms` never surface QA titles — the "QA / Testing Roles" digest section will simply stay empty.
