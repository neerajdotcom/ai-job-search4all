# scraper/ — scraping, location/role tagging, liveness

Deep docs for `scraper/job_scraper.py` and `scraper/ats_scraper.py`. The root
`CLAUDE.md` has the high-level pipeline overview; this file loads when you're
working in `scraper/`.

## Job boards
- Adzuna (country from `profile.adzuna_country_code`, free job search API, no credit card) — syndicated aggregator, can be stale.
- Direct employer ATS feeds via `scraper/ats_scraper.py` — Greenhouse, Lever, Ashby, and Workable public APIs. **Off by default** (`ENABLE_ATS_SCRAPING` env flag) since the company lists in that file are a hand-picked, empty-by-default template — populate `GREENHOUSE_COMPANIES`/`LEVER_COMPANIES`/`ASHBY_COMPANIES`/`WORKABLE_COMPANIES` with your own target employers' board slugs before enabling. Always reflects each company's live open-positions list, so the usual `STALE_DAYS` staleness filter doesn't apply to these.

## Scraper (`scraper/job_scraper.py`)
`scrape_all_jobs(profile)` merges Adzuna (queried once per term in `profile.search_terms`) with direct ATS feeds from `scraper/ats_scraper.py` when `ENABLE_ATS_SCRAPING=true` (`scrape_all_ats_jobs()` — Greenhouse + Lever + Ashby + Workable), dedupes by URL and by title+company, drops anything matching `profile.excluded_companies` (e.g. your current employer), and applies `STALE_DAYS` (7) staleness filtering to Adzuna jobs only — ATS feeds return each employer's current live list, so an old `first_published`/`createdAt` doesn't mean the role is closed. Location is checked twice for every job: once against the structured location field, then again against the *full* JD text — for Adzuna that's the detail-page text fetched via `fetch_full_description` (cached on the job dict as `_full_text` so the scorer can reuse it without a second fetch); for ATS jobs it's the `jd_text` already returned by the same API call, no extra fetch needed. The page-text recheck is authoritative because a job board's own structured `area`/`location` fields can mislabel a posting's real country. Every kept job is also **tagged, not dropped**: `is_qa_role` (via `_is_qa_role()`/`_QA_TITLE_PATTERNS`) and `is_target_location` (via `_is_target_location()`, a permissive-by-default detector comparing the job's location against `profile.target_location_country`/`target_location_aliases`, with a hard drop only for anything matching `profile.blocked_locations`) — these tags drive both the prefilter's scoring priority and the digest's section bucketing, so QA roles and out-of-target-location roles still get scraped/scored, just routed differently downstream instead of being filtered out at scrape time. A third tag, `is_target_role` (via `_is_target_role()`/`_target_role_patterns(profile)`, derived from `profile.search_terms` and `profile.title`, word-boundary regex so it doesn't false-match substrings), exists because ATS feeds return each employer's **entire** job board unfiltered by title (unlike Adzuna, which is queried per `profile.search_terms`) — without it, an unrelated role can land in the scoring batch and, if scored, pollute skill-gap aggregation with irrelevant "missing keywords." This tag doesn't affect scraping or digest routing, but it does gate scoring (see the off-target skip in `scorer/CLAUDE.md`) and also scopes the skill-gap report.

## Location matching (`_is_target_location`, `_is_blocked_location`)
Country-agnostic by design: `_KNOWN_COUNTRIES` is a set of country names the detector can recognize in free text, and `_CITY_COUNTRY_MAP` maps major cities to their country so a target country's own cities are never mistaken for "non-target" (and so an unrelated country's city is correctly recognized even without an explicit country name in the text). `_is_target_location(area, location, profile)` compares the job's location fields (and, via the scraper's callers, the full JD text) against `profile.target_location_country` + `profile.target_location_aliases` (e.g. `"remote"`), defaulting to a match when the location is ambiguous, empty, or plainly remote — permissive by design, since a false "not my target location" silently costs the user a real match. `_is_blocked_location(area, location, profile)` is a separate, harder check against `profile.blocked_locations` (e.g. countries with employer visa-sponsorship restrictions) that drops a job outright rather than just tagging it — leave `blocked_locations` empty to disable this entirely.

## crawl4ai web-board sources (`scraper/crawl4ai_scraper.py`)
Optional additive source feeding `scrape_all_jobs()` for large aggregator boards (Naukri, Indeed, Foundit, Instahyre, Wellfound — no LinkedIn, login wall + account-ban risk). Uses [crawl4ai](https://github.com/unclecode/crawl4ai) (Playwright/headless Chromium) to render each board's JS search page to clean markdown locally — the *fetch* needs no API key. **Table-driven by design** (per the `minimal-code-review`/ponytail ladder): one `SITES` config list (source, URL template, `term_sep`, CSS schema) + one generic `_crawl_site()` loop, **not** five near-duplicate functions — adding a board is a new dict entry. **Hybrid extraction:** free per-site CSS selectors (`JsonCssExtractionStrategy`) first; if a page yields zero rows, one Groq pass over the markdown (`_groq_extract`) as fallback — and it **reuses scorer.match_scorer's `_generate_with_retry` + `_daily_quota_exhausted` + throttle**, so extraction shares the same per-run Groq budget as scoring rather than opening a second LLM path. Produces the **exact ATS job-dict shape** (`title/company/location/jd_text/apply_url/source/posted_at`), so the scorer/optimizer/digest need no changes; `posted_at` is `None` (listing pages rarely expose a reliable date). **Gated behind `ENABLE_CRAWL4AI`** (env flag; default off, **also off in the GitHub Actions workflow** — see the caveat below: these boards commonly hard-block the CI datacenter IP, so it's kept for local/residential-IP runs via `ENABLE_CRAWL4AI=true python main.py`) and **fully fail-open** at every layer — crawl4ai not installed, browser launch failure, a datacenter-IP block, or a parse error each just yield `[]`, never breaking the Adzuna + ATS path. Unlike the live ATS boards, crawl4ai sources are **freshness-filtered like Adzuna** (they're crawled listings, not a live open-positions API) — see `_LIVE_BOARD_SOURCES` below.

**Honest caveat:** these boards typically prohibit scraping in their ToS and actively block automation. GitHub Actions runs on datacenter IPs that many boards block first, so yields from the scheduled CI run may be low or zero even though the same code works from a residential IP — this is expected and non-breaking (the normal digest still ships). The Groq fallback is what carries a board until its CSS selectors are tuned against a real successful fetch. crawl4ai's Chromium is installed best-effort in the workflow (`playwright install --with-deps chromium || true`).

## Liveness re-check (`scraper.job_scraper.check_job_live`)
Right before the digest is built, `main.run()` re-verifies each *qualifying* posting is still reachable (a job fresh at scrape time can be filled days later — `STALE_DAYS` is a one-time scrape-time check). Bounded — runs only on the small qualifying set, skips live-board ATS jobs (an entry existing on a Greenhouse/Lever/Ashby/Workable board already means it's open), and only re-fetches Adzuna links. Fails open: returns `False` (→ `live=False` on the job) **only** on an unambiguous HTTP 404/410; a 200, redirect, timeout, or any error is treated as live. Dead postings are **flagged, not dropped** ("⚠ Link may be closed" in the digest table and on the dashboard), so the user still sees the match.

## Gotchas
- `_LIVE_BOARD_SOURCES` (in `job_scraper.py` = `{greenhouse, lever, ashby, workable}`) is the single source of truth for "which sources are a live open-positions API and thus exempt from `STALE_DAYS` freshness filtering." Everything *not* in this set (Adzuna **and** the crawl4ai web boards) is a syndicated/crawled listing and **is** freshness-checked. The location/full-text-recheck branch keys on `source == "adzuna"` specifically (only Adzuna needs the extra detail-page re-fetch; ATS and crawl4ai sources already carry their full `jd_text`).
- `_is_target_company()` in `job_scraper.py` tags each job `target_company` (bool) from `profile.target_companies` (a hand-maintained list you fill in), alongside `is_qa_role`/`is_target_location`/`is_target_role`. It is **presentation-only** — it drives a "⭐ target company" highlight in the digest/dashboard and has no effect on scoring, qualification, or ordering (see `scorer/CLAUDE.md`). Short/ambiguous company names should use the fuller name as the alias and match as a whole token so they don't false-match unrelated firms.
- `_QA_TITLE_PATTERNS` in `job_scraper.py` matches QA/testing roles by title (QA Lead/Manager/Engineer, Quality Assurance/Engineer/Analyst, Test Engineer/Lead, Tester, SDET, etc.) — word-boundary regex so e.g. "qa" doesn't match "equation". These roles are **tagged** `is_qa_role = True`, not dropped: they're still scraped/scored (and deprioritized for Groq's scoring cap by `_priority_rank` in `match_scorer.py`) and land in the digest's "QA / Testing Roles" section instead of the main table. This QA-specific carve-out is a legacy convenience from the original profile this tool was built for — remove/replace `_QA_TITLE_PATTERNS` if it isn't relevant to your target roles; it costs nothing to leave in place either way (a profile whose `search_terms` never surface QA titles will simply never trigger it).
