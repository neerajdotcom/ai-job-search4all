## Optimization (`optimizer/resume_optimizer.py`)
Runs on **Gemini** (`gemini-2.5-flash`), capped at `AI_OPTIMIZE_LIMIT` (15) per run since Gemini's free tier has a much stricter rate limit (throttled to one call every 13s). Only ever invoked for jobs that already passed the Groq threshold (and survived the reviewer pass) — Gemini's own internal `match_score` is informational only, not a second gate (see the docstring on `optimize_resume`: Groq and Gemini score independently and can disagree, which used to silently drop a resume for a job already shown as qualifying in the digest). It rewrites only the Professional Summary, Core Competencies, and up to 3 bullets per role, preserves protected metrics verbatim, and writes per job into `outputs/`: one patched DOCX (`*.docx`, the editable companion → `review_output`) which is then rendered to a **PDF** (`*.pdf`, the submittable deliverable → `ats_output`) via headless LibreOffice (`_render_pdf`), plus cover note variants and screening Q&A — always working from a copy, never touching the source resume. The single Gemini call (no extra cost) also picks a **framing archetype** for the JD from `profile.archetypes` and biases the rewrite toward it (`chosen_archetype` on the job), and returns `cover_note_variants` — 2-3 angle-tagged cover notes (`domain-fit` / `delivery-track-record` / `growth-story`) the user picks between instead of one fixed draft (a `cover_note` backward-compat alias = `variants[0].text` is kept so older consumers don't break). The PDF is what gets attached to the digest email and offered as the primary download; the DOCX is the hand-editable fallback. If LibreOffice isn't installed or the conversion fails, `_render_pdf` returns `None` and the DOCX itself becomes the submit file (a `quality_warnings` entry records it) — the run never breaks over a missing renderer. `patch_docx` also logs (and surfaces in `quality_warnings`) any section it couldn't locate to patch — Professional Summary, Core Competencies, or a role's bullet block — and re-checks protected metrics against the patched DOCX's actual text before rendering, instead of silently leaving a gap. `quality_warnings` is threaded through `main.py` onto each scored job and is visible on the job detail page in the dashboard.

## Protected metrics — derived per résumé, not hardcoded
`extract_protected_metrics(text)` scans a résumé's own text (percentages, dollar amounts, `Nx` multipliers) and returns whatever quantifiable figures it actually contains — there's no fixed list. `call_llm` extracts from the source résumé text to tell Gemini which figures must survive the rewrite; `patch_docx` extracts from the *original* (pre-patch) DOCX at `source_path` so the "never drop a metric" guard and the post-patch verification both check against the same candidate-specific set, whatever numbers that particular résumé happens to have. This means the guard adapts automatically to any user's résumé with zero configuration.

## Role-bullet patching — driven by `profile.roles`
`patch_docx`'s `role_map` is `profile.roles` (an ordered dict of role-key → match-keywords the user fills in during onboarding, one entry per employer in their résumé) — not a hardcoded company list. The `optimized_bullets` keys requested in `call_llm`'s prompt come from `profile.roles.keys()`, so the two always stay in sync automatically. Bullet-paragraph detection doesn't rely on literal bullet characters or paragraph style names (many résumé templates render bullets via Word's numbering XML, invisible to `python-docx`'s `.text`, with every paragraph styled "Normal") — instead it compares each paragraph's bold/font-size against the "Professional Experience" section heading's own size: a top-level section heading is bold and at least as large as that reference size, a role's internal bold sub-heading is bold but smaller, and bullets are the non-bold paragraphs in between. If a located role section yields zero bullet paragraphs, `patch_docx` now records a `quality_warnings` entry instead of silently leaving it unpatched.

**Template-robustness details (all triggered by a real production run against a Kolkata PMO candidate whose CV template differed from the sample):** (1) `_find_paragraph_index(doc, "Experience")` runs three passes — exact match on stripped text, then starts-with on a paragraph ≤ 60 chars, then substring — because a naive substring search matched the query inside a Professional Summary body ("...Cross-Functional Program Coordination. Experience..."), landing on the wrong paragraph. (2) The Core Competencies heading is tried in five variants (`Core Competencies` → `Competencies` → `Skills Summary` → `Key Skills` → `Skills`) so a CV that names the section differently still gets patched. (3) `_is_section_heading` first checks a well-known-section text list (`professional experience`/`experience`/`education`/`academics`/`skills`/`skills summary`/…) before falling through to the size compare; the old `size is None or sh_size is None → True` fallback was over-aggressive and misclassified every bold role sub-heading as a section boundary, silently losing every bullet under that role. (4) Role-heading lookup requires a **bold** paragraph match first, only falling back to any-paragraph match if none exists — this stops the tailored Summary body (which typically mentions the candidate's current employer by name) from being misidentified as the role heading after the Summary is patched earlier in the same call.

## Gotchas
- `profile.archetypes` (a dict of `{name: description}`) biases word choice only in the optimizer's rewrite — it never changes which sections get patched. A profile's default is a small set of generic framings; replace it with framings that suit your own target role mix.
- Resume PDFs are rendered by **headless LibreOffice** (`soffice`/`libreoffice`), invoked via `subprocess` in `resume_optimizer._render_pdf` — no Python package, but it is a *system* dependency. It's preinstalled on GitHub's `ubuntu-latest` runners (the workflow also has a best-effort `apt-get install libreoffice-writer` guard) and on most dev machines. Absence is non-fatal: `_render_pdf` falls back to the DOCX and records a `quality_warnings` entry. Dry runs never render (optimize is skipped), so LibreOffice isn't needed for `--dry-run`.
- Same `_MIN_INTERVAL` throttle / `_daily_quota_exhausted` short-circuit pattern as the scorer — see `scorer/CLAUDE.md`.
