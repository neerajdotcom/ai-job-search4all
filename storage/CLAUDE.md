# storage/ — git-committed JSON persistence ("git is the database")

Deep docs for `storage/run_store.py` (immutable per-run snapshots) and
`storage/tracker_store.py` (mutable cross-run tracker / Career CRM). Loads
when working in `storage/`. High-level overview is in the root `CLAUDE.md`.

## Persistence model — git-committed JSON, not a database
Every pipeline run (CLI, GitHub Actions, or dashboard-triggered) is saved as one file at `data/runs/<run_id>.json` (`run_id` = `started_at` formatted `%Y%m%dT%H%M%SZ`) by `storage/run_store.save_run_snapshot()`, called from the end of `main.run()`. This is intentional — no SQLite, no server-side DB, no extra hosting cost; git itself is the database, so a run snapshot committed by a scheduled GitHub Actions run is immediately visible to the dashboard next time you `git pull` locally, and vice versa. `data/runs/*.json` is **not** gitignored (unlike `outputs/`) — it's meant to be committed and accumulate as history. Each snapshot holds the full run summary plus every *scored* job, qualifying or not (needed for the score-distribution chart), with `jd_text`/`_full_text` stripped before writing (large, no dashboard value, would bloat git history). `storage/run_store.py` also exposes `list_runs()` (summaries only, for table/list views), `load_run(run_id)` (full snapshot incl. `jobs[]`; validates `run_id` against a strict filename regex to block path traversal from the URL param), and `compute_stats()` (aggregates across all snapshots for the home page's charts).

## Cross-run dedup tracker / Career CRM (`storage/tracker_store.py`, `data/tracker.json`)
Unlike `run_store`'s immutable per-run snapshots, this is one mutable map keyed by cleaned apply URL, holding each posting's full lifecycle state. It exists for the per-job question run snapshots can't cheaply answer — "have I already scored this exact posting on a previous day?" — and powers the pre-scoring dedup gate, the ghost-job/analytics/weekly-report aggregations, and the dashboard's application-status dropdown and bookkeeping fields. Also git-committed (not gitignored).

Before the prefilter ranking (and well before any Groq scoring), `filter_unscored()` drops postings already scored within `RESCORE_TTL_DAYS` (14) on a previous run, so the same Scopely/Riot/Epic postings don't re-spend a scarce `AI_SCORE_LIMIT` Groq slot day after day. This has to run on the full scraped pool, not a pre-cut top-N — running it after the prefilter (the original order) meant dedup only ever saw the same ~50 keyword-top jobs and had nothing left to skip, silently starving the pipeline of new candidates for days. State is keyed by `_clean_url(apply_url)` with a `title|company` fallback — reusing the scraper's own `_clean_url`/`_normalize_for_dedup` so "same job" means the same thing as the within-run `_dedup`. `main.run()` calls `mark_seen(all_scored)` mid-run (after scoring/optimizing, before the digest is built, so the weekly report and digest both see this run's fresh state) to record/refresh each scored posting's `first_seen`/`last_seen`/`last_scored`/`last_score`/`run_count`/`source`/`industry_fit`/`seniority_fit`/`missing_keywords`/`resume_version` (the optimized DOCX/PDF filename, if one was produced)/`chosen_archetype`/`ghost_flagged`/`ghost_reason`, plus a `status` (`new → scored → qualifying → optimized → applied → …`) and an append-only `status_history` ({status, at} list, written on every transition, automatic or manual). The auto statuses (`AUTO_STATUSES = {new, scored, qualifying, optimized}`) are advanced by the pipeline; the manual ones — `MANUAL_PROGRESSION = [applied, interview_scheduled, interview_complete, offer]` plus terminal `{rejected, withdrawn, archived}` and legacy aliases `{responded, interview, closed}` kept valid for backward compatibility — are set from the dashboard via `set_status()`, which rejects backward moves within `MANUAL_PROGRESSION` (e.g. `offer → applied`) and is never downgraded by a later automatic `mark_seen`. A job scored from a quota-exhausted fallback (`"quota exhausted"` in its `recommendation`) only gets `first_seen`/`last_seen` refreshed — `last_scored`/`status` are left untouched so `filter_unscored` still re-offers it for scoring next run instead of wrongly treating a never-evaluated job as scored for the full TTL. `recruiter_name`/`notes`/`follow_up_date` are manual-only bookkeeping fields, untouched by `mark_seen`, set from the dashboard's job-detail page via `set_fields()`. `export_csv()` flattens the tracker into `data/tracker_export.csv` (gitignored — a derived spreadsheet view regenerated each run, not a second source of truth; `data/tracker.json` stays canonical and git-committed) with one row per posting (`date_discovered`, `last_seen`, `company`, `role_title`, `match_score`, `resume_version`, `application_status`, `apply_url`, `source`, `missing_keywords`, `industry_fit`, `seniority_fit`, `follow_up_date`, `recruiter_name`, `notes`), called once at the end of `main.run()`. `--no-dedup` bypasses the scoring-skip gate (but not the tracker writes) to force a full re-score.

## GitHub Actions persistence step
The `run-pipeline` job has `permissions: contents: write` and a "Commit run snapshot" step (after "Run pipeline", before the artifact upload) that `git add`s `data/runs/` and `data/tracker.json`, commits with `[skip ci]` only if there's something new to commit (`git diff --staged --quiet` guard avoids empty commits when a run produced no new snapshot, e.g. it crashed before reaching the save step), and pushes using the default `GITHUB_TOKEN`. This step runs `if: always()` and never affects the job's pass/fail status; `notify-failure` is untouched. On a Friday run it also stages `data/weekly_report_*.md`.

## Gotchas
- `data/tracker.json` (cross-run dedup/status) is committed like `data/runs/`. A previously-scored posting is skipped for `RESCORE_TTL_DAYS` (14) — so a qualifying job surfaces in the digest the day it's first scored, then not again until the TTL lapses or the JD reappears as a new URL. Use `--no-dedup` to override.
- `data/tracker_export.csv` is a derived view of `data/tracker.json`, regenerated by `export_csv()` every run and gitignored — `data/tracker.json` is the only source of truth for tracker state; never write CSV-only fields back into it. `data/weekly_report_*.md`, by contrast, **is** git-committed (like `data/runs/` and `data/tracker.json`) since it's a point-in-time report, not a derived cache — only ever written on a Friday-in-IST run (`is_report_day()`), so its absence on any other weekday run is expected, not a bug.
